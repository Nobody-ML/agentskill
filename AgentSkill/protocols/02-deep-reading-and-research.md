# 02-deep-reading-and-research：深读与调研协议

适用范围：
- Track：Research / Software / Writing
- Level：L2 / L3（强制）

目标：
- 把“我看过了/我查过了”变成可复查的证据：读了什么、结论是什么、对方案的影响是什么。
- 让调研服务于“可落地方案 + 可验证计划”，而不是堆链接。

相关门禁：
- `protocols/00-hard-gates.md`：G2（Deep Reading）/ G3（Research）
- `protocols/01-plan-mode-and-deep-plan.md`：Deep Reading / Research Log 标准

---

## 1) 深读（Deep Reading）执行顺序

深读的推荐顺序（按“离用户目标最近”优先）：

1) **用户材料**
   - 需求描述、失败样例、约束、偏好、验收、禁区
2) **现有工件**
   - `State.md`（边界/授权/Next Action）
   - 当前版本 `Plan.md` / `Task.md`
3) **仓库地图（针对工程任务）**
   - 入口点（CLI/服务入口）
   - 核心模块边界与依赖
   - 关键数据结构与协议
   - 现有测试、样例、fixtures、golden outputs
4) **现有验证与证据**
   - 测试报告、日志、截图、基准、实验记录
5) **外部事实与一手资料**
   - 官方文档、标准、论文、源码

硬规则：
- 深读必须落盘在 Plan 的 “Inputs and Materials Read” 中，并说明“读到哪里 + 关键结论 + 对方案的影响”。

## 2) 外部检索（Web Research）质量规则

### 2.0 强制性（何时必须联网检索）

默认规则：
- Level = L3：只要任务涉及第三方库/工具/标准/内部格式/版本兼容，**必须**联网检索并落盘来源（除非用户明确禁止联网）。
- Level = L2：当出现以下任一信号，视为必须联网检索：
  - “最新/当前/版本变化/兼容性/许可证/合规”
  - 需要选型或对比多个技术路线
  - 涉及复杂格式/交互保持（例如 PDF/排版/字体/坐标/链接/目录/书签/注释）
  - 需要引用支撑关键结论（否则结论只能写成待证实假设）

用户明确禁止联网时：
- 研究门禁（G3）状态必须为 `blocked`
- Plan 必须把“缺失的外部事实”写为阻塞点（Blockers）并列出最小输入（例如：用户提供文档/版本信息/内部资料）

重要约束：
- 不需要用户显式说“请联网检索”。当任务命中上述信号时，联网检索是默认必做项。

优先级：
1) **一手资料**：官方文档、标准、论文、源码
2) **权威二手**：教材/权威综述/同行评审后的总结（需要交叉验证）
3) **社区内容**：只能当线索，必须找到一手佐证才能写入结论

落盘要求（写入 Plan 的 Research Log 或单独 ResearchLog）：
- Query：检索关键词
- Source：来源与链接
- Extracted facts：抽取事实（短句）
- Impact：影响了哪条路线/哪个验收/哪个风险
- Open questions：仍不确定的点
- 关键事实必须能“点开验证”：Source 需要是可访问的链接或可定位的一手材料入口

禁止：
- 无来源硬写结论
- 把“印象/常识”当成事实引用
- 用“用户未要求联网/未要求外网检索”作为跳过 Research Gate 的理由

### 2.1 最低研究包（L3 默认要求）

当任务属于 L3 且需要外部事实时，Research Log 至少包含：
- **≥ 8 条来源**（可更多）
- 至少 **≥ 3 条一手资料**（官方文档/标准/论文/源码）
- 至少 **≥ 1 条“源码/issue/实现细节”证据**（用于确认边界行为与坑点）

并且每条来源必须写清：
- 该来源解决了哪个关键问题（对应 Plan 的某一段）
- 该来源如何改变方案/验收/验证（Impact）

### 2.2 版本与许可证（不许省略）

对第三方库/工具必须记录：
- 版本或日期（避免把过期信息当成事实）
- 许可证与风险点（尤其当目标是可分发/可商业化的软件）

这两项必须能指向来源入口，不得凭印象写结论。

### 2.3 外科手术型任务（PDF/排版/交互保持）研究清单（强制）

当任务命中“格式/交互/排版保持”的外科信号（Router L3）时，Research Log 至少要覆盖这些主题（每条都要有来源与 Impact）：

- 解析器/版面工具的输出结构与版本差异（例如：layout JSON schema、bbox 表示、breaking changes）
- 坐标系与旋转页：内部统一坐标约定、必要的坐标转换与验证方式
- 文本删除/重写的副作用：删除文字是否会影响 link/annotation/outline 等功能对象
- 低层 PDF 对象：加密/权限、命名目标、页面标签、附件/元数据、线性化与修复（按需）
- 字体与中文排版：缺字、字体替换、文本测量与 fit 策略
- 渲染 QA：至少两种渲染引擎/阅读器差异的风险与抽检策略
- 数字签名：修改正文导致签名失效的预期与交付说明要求

对应 playbooks：
- 通用：`library/format-surgery-systems.md`
- PDF 专用：`library/pdf-layout-translation-playbook.md`

## 3) 证据管理（Evidence Management）

目标：
让验证与交付能复跑、能审计。

要求：
- 每次关键结论必须能定位到证据入口（文件路径/命令/截图/日志/链接）
- Evidence Index 必须写入 `State.md`

推荐结构：
- Evidence 按层次组织：micro-check / checkpoint / milestone / final
- 证据文件命名带时间与范围（例如 `reports/checkpoint-2026-04-20.json`）

---

## 4) 研究任务的额外要求（Research Track）

当任务目标是“提出/审查/验证 idea”时：
- 必须给出可验证的实验/消融计划（指标、数据、对照组、失败判据）
- 必须给出可复现协议（参数、环境、随机性控制、脚本入口）
- 文献检索要覆盖“最相关的主线 + 反例/负结果/局限”

- `templates/Literature-Review-Matrix.template.md`
- `templates/ReproProtocol-Research.template.md`
