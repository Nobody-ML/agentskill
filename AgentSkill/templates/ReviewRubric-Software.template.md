# Review Rubric（Software）— 评分与整改模板

标度：0–10 分（允许小数）

锚点解释：
- 9–10：优秀；正确、清晰、可维护、验证充分，风险可控
- 7–8：良好；可合并/可交付，少量改进即可
- 5–6：可用但不稳；关键测试或边界处理薄弱
- 3–4：问题较大；实现/设计偏离需求或质量明显不足
- 0–2：不可用；不正确/不安全/不可验证

---

## 1. 硬门禁（先过门禁再谈分数）

- **过程合规违规（Stop-the-line）**
  - 场景：未授权就执行（Execution Authorization!=received 但发生改动/运行）、Plan/Task 契约缺失或残留大量占位符、关键改动无法映射到 Task 条目、或真数据门禁违规
  - 结论必须为：`BLOCKED`
  - 动作：回到 Plan 补齐 Plan/Task/State 与证据门禁，再重新走授权门禁

- **验证失败（failed）**（测试失败 / typecheck 失败 / lint 失败 / 手动路径跑不通）
  - 结论必须为：`BLOCKED`
  - 动作：必须返工修复 → 再复评

- **验证缺失（missing）**（项目无测试 / 环境不可运行 / 无法获得证据）
  - 结论默认应为：`NEEDS_USER_DECISION`
  - 动作：必须提醒用户风险；是否返工补齐验证由用户决定（默认建议补齐最小验证）

## 3. 总分与结论

- 总分（建议取均值或加权均值）：

结论枚举：
- `PASS`：验证通过且总分 ≥ 7.0/10
- `NEEDS_USER_DECISION`：
  - 验证通过但总分 < 7.0/10（需提醒风险；是否返工由用户决定）
  - 或验证缺失（missing）：必须提醒风险；是否返工由用户决定
- `BLOCKED`：验证失败（failed）（必须返工并复评）

> 用户偏好规则（要执行）：
> - 测试/验证不通过 → 必须复评
> - 总分 < 7.0/10 且测试不通过 → 必须返工并复评
> - 总分 < 7.0/10 但测试通过/或无测试可做 → 必须提醒用户；是否返工复评取决于用户反馈
