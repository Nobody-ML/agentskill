# Plan Mode Interaction（对齐与提问收敛）检查表

适用：Level ≥ L2 的大任务（Research / Software / Writing），或任何“需求不清/方法不稳/风险较高”的任务。

目的：让对齐阶段真正变成“可执行的收敛过程”，而不是反复聊天或在信息不全时直接开做。

---

## 触发条件（任一命中就启用）

- Level ≥ L2
- 需求完整性快评 < 7/10
- 验收标准写不出来
- 技术选型/架构取舍会显著影响成本与风险
- 需要真实数据验证，但数据来源不明确
- 存在安全/合规/不可逆风险

---

## A. Plan Mode 的允许动作与禁区

允许：
- 读材料：用户文档、仓库代码、配置、日志样本（只读）
- Research：检索高质量来源并落盘（来源可追溯）
- 落盘工件：Plan/Task/（必要时）Acceptance Contract、验证矩阵、Runbook 草案

### A.1 深读与检索门禁（防止“读了但没读到东西”）

深读最低要求（L2/L3 默认执行）：
- 用户材料：提炼出约束/验收/接口/数据/失败判据，并写入 Plan 的 Read Log
- 代码库：定位关键入口/关键数据流/关键失败路径/关键验证入口（测试/样例），并写入 Read Log

检索最低要求（命中即强制）：
- 涉及选型/用库/对标/“最新”结论：必须做高质量检索，并写入 Plan 的 Research Log（来源 + 关键结论 + 对方案的影响）

禁止（在未授权 `开始执行` 前）：
- 改动目标代码/配置/生产环境
- 除 Plan/Task/State/治理模板外，修改任何“交付物”（代码/脚本/配置/文档/实验产物）
- 进入长时间训练/长时间运行
- 任何破坏性操作（删除、覆盖、迁移）

---

## B. 提问收敛规则（少而关键）

硬规则：
- 每轮问题 **最多 3–6 个**
- 每个问题必须满足：回答会改变 **验收标准/范围边界/方法选择/验证方式** 之一
- 能从材料推断的，不反问

推荐提问形态（收敛式）：
1) **确认断言**：把已推断的内容写成一句可验收断言，请用户确认/纠正
2) **边界确认**：用“做/不做”列出边界，请用户确认
3) **资源确认**：用具体量级问（时间/算力/数据权限），避免抽象问法
4) **验证入口确认**：真实数据/样本/日志是否可提供（可脱敏、可裁剪）

---

## C. 默认值策略（避免对齐阶段卡死）

当信息缺失但不属于高风险决策：
- 给出一个“推荐默认值”
- 明确写入 Plan/State 的 Assumption
- 在 Task 中安排一个“验证/澄清”任务，把不确定性变成可执行动作

当信息缺失且属于高风险决策：
- 不猜；写成阻塞点（P0）
- 明确需要的最小输入（例如：一份脱敏样本、一条复现命令、一个权限开关）

---

## D. 对齐完成的判据（退出 Plan Mode 的条件）

退出 Plan Mode 必须同时满足：
- [ ] 目标明确（Goal 一句话可复述）
- [ ] 至少 2 条可验收标准（可观察/可测试/可复现）
- [ ] 边界明确（Non-goals 清晰）
- [ ] 验证方案明确（Validation Plan + 证据入口）
- [ ] Task 拆分到可勾选最小单元（每条都有产出与验收方式）
- [ ] 执行授权门禁就绪（Execution Authorization=required，等待用户回复 `开始执行`）

---

## E. 执行授权（Plan Mode → Execution Mode）

规则：
- L2/L3 默认 `Execution Authorization = required`
- 为避免误判：只有当用户**单独回复** `开始执行`（建议独立消息或独立一行）后，才进入 Execute/Write
- 授权必须落盘：time + scope（允许执行的范围/目录/环境）

证据落盘位置（推荐）：
- Plan.md：Execution Authorization 段落
- State.md：Execution Authorization 段落 + Decision Log（如需）
