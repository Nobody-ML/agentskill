# Development Paradigms（开发范式协同）作战手册

适用：Software Track（尤其 Level ≥ L2），以及 Research/Engineering 混合任务（要写代码 + 要验证）。

目标：把“开发方法论”从口号变成可选择、可落盘、可验证的工具箱；避免把任何单一范式当成万能钥匙。

---

## 1) 工具箱（不是单选题）

常用范式/套路（可组合）：

1) **SDD(spec-d-d)（Spec-Driven Development / 规范驱动）**
   - 适用：大多数工程任务（尤其 L2/L3）；需要“可验收、可复查、可复现”的交付
   - 最小步骤：
     1. 把需求写成“可执行规范”：验收标准、示例输入输出、错误语义、数据约束、不变量、边界与失败判据
     2. 先搭最小验证骨架（自动化断言/脚本/验收契约映射），再写实现
     3. 实现→重构→补齐规范覆盖（规范与证据同步落盘）
   - 主要收益：把“做对/做完”的定义前置；减少返工；证据链更完整

2) **Characterization Tests（特征测试 / 行为锁定）**
   - 适用：接手遗留代码、行为不清、没有测试但又必须修改
   - 最小步骤：先把现有行为“测出来并固定” → 再改实现 → 用固定的测试防回归
   - 主要收益：降低改动恐惧；把隐性行为显性化

3) **BDD（验收/行为驱动）**
   - 适用：需求以用户行为为中心、验收断言需要产品/测试共同确认
   - 最小步骤：先写验收断言（Given/When/Then 或 AC-XXX）→ 再实现
   - 主要收益：验收更清晰；减少“做完才发现不是要的”

4) **Spike / Prototype（探索性原型）**
   - 适用：高不确定（新框架/新库/新算法/性能未知）
   - 最小步骤：先做最小原型验证关键假设（可丢弃）→ 再进入工程实现
   - 主要收益：尽早证伪；避免在错误方向上堆工程量

5) **Contract-first / Interface-first（契约/接口先行）**
   - 适用：模块边界清晰、多人协作、未来需要替换实现
   - 最小步骤：先定义接口与错误语义 → 再实现 → 再补集成验证
   - 主要收益：边界更干净；变更成本更可控

6) **Observability-driven Debugging（可观测性驱动排障）**
   - 适用：线上问题、黑盒系统、性能/稳定性问题、训练监控与长任务运行
   - 最小步骤：先补观测点（logs/metrics/trace/关键产物）→ 再复现与缩小范围 → 再修复 → 再验证
   - 主要收益：避免“盲改”；定位效率高；证据可复查

## 2) 选择表（按场景推荐组合）

| 场景 | 推荐组合 | 不推荐做法（风险） |
|---|---|---|
| 新功能/新模块（greenfield） | SDD（先规范后实现） + Contract-first（边界清晰时） + 最小集成验证 | 先搭大框架后补验收与证据 |
| 修 bug（可复现） | 复现脚本/回归断言（或 characterization）+ 最小修复 + 回归 | 先改代码再想怎么证明修好了 |
| 遗留系统改动（legacy） | Characterization → 小步改动 → 逐步补测试 | 大重构 + 无证据迁移 |
| 性能退化/稳定性 | Observability-first + 复现/基线 + 定点优化 | 只靠直觉改热点 |
| Ops/监控/训练盯盘 | Runbook + 观测点 + 阈值 + 证据落盘 | 只盯屏幕，不记录可复查证据 |
| 研究型实现（算法/实验） | Spike/MVE → 固化脚本与参数 → 复现协议 | 先写完整工程再跑实验 |

---

## 3) 落盘规则（把“方法论”写进工件）

在 Level ≥ L2 的任务中，Plan.md 至少写清：
- 本次采用的范式组合（Development Strategy）
- 为什么选它（结合风险与约束）
- 该组合如何产生证据（对应 Validation Plan）

在 Review 时必须能回答：
- 选用的方法是否降低了风险？
- 证据是否足够支撑“完成/通过”的声明？
